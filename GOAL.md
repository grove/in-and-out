# Goal: Declarative MDM Synchronization Tools ("In-and-Out")

This project aims to research and build two separate but related tools that act as the automated ingestion and output layers of a Master Data Management (MDM) system, built on top of PostgreSQL 18.3.

## The Strategy
- **Declarative Integrations:** Instead of writing code for every new integration, the tools interpret configuration files (e.g., YAML/JSON) to define how data is read or written.
- **Shared HTTP Logic:** Since most external systems communicate via HTTP APIs, the underlying execution engine will share HTTP client logic (auth, pagination, retries) across both tools.
- **Sync Modes:** The primary mode of operation is **Incremental Synchronization**, utilizing high-water marks (e.g., `updated_at`). However, both tools must gracefully handle **Full Synchronizations** when external APIs lack mechanisms to query incremental changes or during initial loads.
- **Infrastructure-as-Code Compatibility:** Connector configuration files must be designed for IaC workflows — version-controllable, diff-able, and renderable via templating tools. This rules out binary or opaque formats and ensures configuration can be managed alongside application code in version control.
- **Runtime Control Table:** Both tools expose a dedicated PostgreSQL control table through which operators can issue commands at runtime — triggering re-syncs, pausing processing, adjusting rate limits, or initiating credential rotation — without restarting the running process.
- **Simulator-First Testability:** The connector abstraction must make it straightforward to swap a real external system for a lightweight simulator (a configurable HTTP stub/mock server) without changing any engine code. Simulators must be usable in CI/CD pipelines and local development where access to live external systems is unavailable, impractical, or undesirable. Each connector's test suite should ship with a reference simulator covering its key interaction patterns (pagination, auth flows, webhooks, error conditions, rate-limit responses).
- **Execution Model:** Both tools run as long-lived daemon processes, not short-lived tasks. The ingestion tool maintains a persistent HTTP server for webhook reception and manages scheduled polling loops internally. The writeback tool maintains a persistent listener on PostgreSQL logical replication. Each tool must be independently restartable with no data loss on crash or restart.
- **Credential Storage:** Connector credentials and tokens are never stored in the connector configuration files themselves. They are stored in encrypted PostgreSQL columns or injected via environment variables or an external secrets manager (e.g., HashiCorp Vault, AWS Secrets Manager). Configuration files reference credentials by name; the runtime resolves the actual values.
- **Observability:** Both tools must emit structured operational data — metrics, logs, and traces — suitable for integration with standard observability stacks. Instrumentation is a first-class design concern, not an afterthought.
- **Graceful Shutdown:** On receiving a termination signal (SIGTERM/SIGINT), both tools must complete in-flight HTTP requests and commit the current page or batch's data before exiting. The process must not drop a partially-fetched page or leave a desired-state batch in an ambiguous committed/uncommitted state on exit. A maximum drain time is configurable; after it elapses the process may exit with a warning, having committed whatever progress has been confirmed.
- **Configuration Hot-Reloading:** When a connector configuration file is updated, the daemon picks up the changes at the start of the next sync cycle for that connector — without a full process restart. Changes must not apply mid-operation. Operators may also trigger an explicit config reload via the runtime control table. Changes that alter database table schemas require a separate migration step and cannot be hot-reloaded silently.
- **Multi-Instance & High Availability:** Running two instances of the same tool concurrently must be a supported deployment topology. Cross-instance concurrency control must use PostgreSQL advisory locks (or equivalent database-backed locks) rather than in-process mutexes, so the guarantee holds across restarts and separate processes. At most one instance may hold the lock for a given connector/datatype pair at a time. This extends the in-process guarantee of requirement T1 #36 to the distributed case.

## Tool 1: The Ingestion Tool
**Objective:** Extract data from external HTTP APIs and synchronize it into PostgreSQL 18.3.

**Key Requirements:**
1. **Per-Datatype Tables:** One dedicated table per distinct datatype discovered in the external system.
2. **Flexible Schema (JSONB):** The database schema avoids brittle structured columns, relying instead on PostgreSQL's `JSONB` to store the native data payload alongside operational metadata. Every per-datatype table must include the following standard columns: `external_id` (source system primary key), `data` (JSONB processed payload), `raw` (JSONB unmodified copy of the source response), `_ingested_at` (ingestion timestamp), `_sync_run_id` (link to the sync run log), `_raw_hash` (SHA hash of the raw payload for change detection), `_deleted` (boolean deletion flag), `_deleted_at` (deletion timestamp), `_schema_version` (integer tracking detected schema changes), and `_source_version` (source system version/etag, when available).
3. **Full & Incremental Modes:** Support incremental fetching when APIs permit, but implement robust full-sync mechanisms (e.g., diffing/hashing existing records against new full payloads) when they do not. The high-water mark used for incremental fetching must be persisted in a dedicated watermark table in PostgreSQL — not in-memory only — so it survives process restarts. An operator may reset the high-water mark for a specific connector/datatype via the runtime control table to force a full re-sync. See requirement #40 for the full watermark specification.
4. **Deletion Tracking:** Detect objects that have been deleted from the source system. During full syncs this is done by diffing against previously known records; during incremental syncs this may rely on deletion events or tombstone markers if the API provides them.
5. **Deletion Verification:** Before marking a record as deleted, perform an explicit targeted lookup against the source system to confirm the object is truly gone. This guards against false positives caused by faulty pagination, transient API errors, or incomplete result sets.
6. **Near Real-Time Ingestion (Event-Driven):** Support webhooks, event streams, or similar notification mechanisms to fetch changes as soon as they occur. Because external systems have varying capabilities, this must gracefully fall back to polling if needed.
7. **Webhook Lifecycle Management:** Actively maintain webhook registrations over time — including initial registration, periodic renewal/re-registration, health checks, and cleanup of stale subscriptions.
8. **Full-State Resolution from Events:** When a webhook or event stream delivers a partial or notification-only payload (e.g., "object X changed"), the tool must perform a follow-up lookup to retrieve the full current state of the object.
9. **Parameterized Sources:** Some APIs return only a list of identifiers or partial/stub objects from their list/search endpoints rather than full records. In these cases the ingestion tool must perform individual follow-up lookups — one per identifier — to retrieve the complete object state before persisting it.
10. **Single Source-of-Truth Table:** Combine webhook/event data with the last known persisted state to produce and maintain one authoritative source-of-truth table per datatype. This table always reflects the best-known current state of every object. When a webhook event and a concurrent poll both produce a state for the same record, the most recently timestamped state wins. If timestamps are equal or unavailable, the webhook-derived state takes precedence as it represents the most recent known change notification.
11. **Declarative Authentication Schemes:** Support a variety of authentication mechanisms declaratively — including OAuth2 (with automatic access token refresh via refresh tokens), API keys (header or query parameter), JWT-based auth, and custom/proprietary authentication flows. Credentials and token lifecycle must be managed automatically without requiring custom code per integration.
12. **Pagination Support:** Handle all common API pagination strategies declaratively — offset-based, cursor-based, and link-header-based (next-page URLs). The configuration must support custom termination strategies (e.g., empty result set, not-full page, missing next-link) to know when to stop fetching.
13. **Circuit Breakers (Sporadic Empty Response Protection):** APIs can intermittently return empty results due to transient failures, restarts, or authentication problems. During a full sync, treating this as "all records deleted" would be catastrophic. The tool must implement circuit breaker mechanisms that halt processing when results are suspiciously empty or significantly fewer than expected, preventing mass false deletions. A circuit breaker must trigger when: (a) a paginated result set returns zero records but the previously known record count exceeds a configurable threshold; (b) a result set shrinks by more than a configurable percentage from the prior run without any accompanying deletion events; or (c) consecutive empty pages are returned before the pagination termination criteria are met. Once triggered, the circuit breaker must be manually reset by an operator — via the runtime control table — after confirming the external system's state is genuinely empty.
14. **Delta-Only Sources:** Some APIs only provide a stream of changes (deltas) and never offer a full snapshot. The tool must handle these by accumulating state locally, never resetting the stream position, since data would be irrecoverably lost.
15. **Change History / Versioned Audit Trail:** Store historical versions of ingested records (or at minimum track what changed and when), not just the latest state. This enables best-effort history reconstruction for systems that don't provide version history natively. The history mode — latest-only (overwrite) vs. full append (one row per version) — must be configurable per datatype. *See requirement #30 for the authoritative storage strategy definition; #15 and #30 are complementary and must be implemented as a single coherent feature, not two separate ones.*
16. **Linked/Nested Object Resolution:** Parent records often embed lists of child object *identifiers* rather than full child objects (e.g., an order containing a list of order-line IDs). The tool must extract those IDs, perform individual lookups to resolve child entities, and persist them into their own dedicated datatype tables.
17. **Raw Data Preservation:** Store a clean, unmodified raw copy of each entity exactly as received from the source system, separate from any derived or enriched representation. This ensures transformations can be re-applied later without re-fetching from the source, avoiding unnecessary load on external systems.
18. **Politeness & Rate Limiting:** The ingestion process must strictly enforce rate limiting, batching, and backoff strategies to prevent accidental Denial of Service (DoS) attacks on external systems, preserving their stability regardless of change volume (e.g., debouncing webhook triggers).
19. **Shared Event Receivers with Declarative Fan-Out:** Some systems (e.g., HubSpot) deliver all webhook events to a single HTTP endpoint regardless of datatype. The tool must support declarative routing rules that filter and dispatch incoming events to the correct per-datatype handler based on event metadata (e.g., event type, account ID).
20. **Multi-Tenancy Support:** A single connector configuration may serve multiple accounts or organizational units within the same external system. All operations — data collection, webhook registration, and event filtering — must be correctly scoped to the active tenant or account at runtime.
21. **Declarative Field / Property Selection:** Some APIs omit fields unless explicitly requested. The configuration must support per-operation field enumeration so the tool requests exactly the right properties, without hardcoding field lists in the engine. The selection syntax should support wildcards and glob-style exclusion patterns (e.g., `!*.internal_field`) for bulk field filtering without enumerating every field individually.
22. **Relationship Datatypes as First-Class Entities:** Many-to-many relationships (associations, links, memberships) in external systems should be treated as independent datatypes with their own ingestion pipelines and dedicated tables, not merely as embedded sub-objects of parent records.
23. **Read-Only Datatype Support:** Some datatypes in a connected system are available for ingestion but have no writable counterpart. The declarative config must support flagging a datatype as ingestion-only so the writeback tool ignores it entirely.
24. **Custom Pre-Request Authentication Flows:** Beyond OAuth2 and API keys, some systems require a dedicated API call to acquire a session token before any other requests can be made. The authentication framework must support arbitrary pre-request token acquisition flows declaratively.
25. **Webhook Event Deduplication:** The same webhook event may be redelivered multiple times due to retries or restarts. The tool must track processed event IDs to prevent duplicate follow-up lookups, duplicate writes, or duplicate state transitions.
26. **Ownership Scoping for Webhook Cleanup:** When deregistering stale or orphaned webhook subscriptions, the tool must only remove subscriptions it created — identified by a known callback URL or tag — never those registered by other applications in the same external system.
27. **Configurable Response Expressions:** The way timestamps, record arrays, and primary keys are encoded in responses varies per system and datatype. These must be declaratively configurable: which field holds the last-modified timestamp, which envelope property wraps the result array, and how to compute composite or expression-based primary keys from multiple response fields.
28. **Connector-Level Runtime Parameters:** Some configuration values (e.g., the public callback URL for webhook registration) are deployment-environment concerns, not integration logic. The connector config format must support first-class runtime parameters that are injected or resolved at deploy time.
29. **Intra-Sync Checkpointing:** A full sync run must checkpoint its progress *within* the run, not only between runs. If a full sync processing millions of records fails at 90% completion, the next run must resume from the checkpoint, not restart from the beginning. Checkpoint granularity should be configurable per datatype.
30. **Selectable History Mode per Datatype:** Each datatype must declare its storage strategy independently: `overwrite` mode maintains only the latest known state (suitable for most operational datatypes), while `append` mode retains a complete version history with one row per ingested version. Both modes must produce a stable "current state" view.
31. **External API Schema Change Tracking:** When the external API's response structure changes — new fields appear, fields are removed, or value types change — the ingestion tool must detect and record this schema drift rather than silently accepting or rejecting it. Schema versions should be stored alongside the data so transformations can be re-evaluated when the schema evolves.
32. **Tombstone Records on Deletion:** When a record is confirmed deleted (after verification per requirement #5), the ingestion tool must write an explicit tombstone record to the per-datatype table — a null or empty payload entry with deletion metadata (timestamp, source confirmation) — rather than performing a hard delete. This makes deletions observable to downstream consumers and change-history queries without requiring access to diff tables.
33. **Intra-Sync Deduplication:** During a single sync run, the same record may appear multiple times — via both a webhook event and the polling full-sync overlapping in time, or via duplicate entries in a paginated result set. The tool must deduplicate records within a run by primary key before writing, ensuring each object is written at most once per sync cycle.
34. **Inbound Webhook Signature Verification:** The ingestion tool must verify cryptographic signatures on inbound webhook payloads before processing them. The verification scheme (e.g., HMAC-SHA256 over the raw body compared against a header such as `X-Hub-Signature-256`) must be declaratively configurable per connector. Payloads that fail signature verification must be rejected with an appropriate HTTP error response and logged, and must never be processed.
35. **Out-of-Order Event Handling:** Webhook events and delta stream entries may arrive out of chronological order — for example, a `deleted` event may arrive before the final `updated` event for the same record. The ingestion tool must compare event sequence numbers or timestamps to detect and discard stale updates. When ordering cannot be determined, the tool must apply a configurable strategy: `accept-latest-timestamp` (default), `accept-highest-sequence`, or `buffer-and-reorder`.
36. **Per-Datatype Concurrency Control:** At most one active sync operation (full sync, incremental poll, or bulk webhook replay) may run per datatype per connector at any given time. Concurrent attempts must either queue and serialize, or the later attempt must abort gracefully with a logged warning. This prevents race conditions where two simultaneous full syncs produce conflicting deletion diffs. The lock must be implemented using PostgreSQL advisory locks so that it is honoured across multiple process instances — in-process mutexes are not sufficient for multi-instance deployments (see Strategy "Multi-Instance & High Availability").
37. **Sync Scheduling:** The ingestion tool must include a built-in scheduling mechanism capable of running per-datatype sync cycles on configurable intervals (cron-style expressions or fixed intervals). The internal scheduler must be externalizable — an operator must be able to disable it and drive syncs entirely through the runtime control table or a process manager without modifying code or configuration.
38. **Pagination Drift Protection:** During a paginated full-sync fetch, records added to or deleted from the external system between page requests can corrupt the result set (the "page drift" problem). The tool must mitigate this by: preferring a server-side snapshot mechanism when the API supports it (e.g., a cursor that freezes the result set at fetch time); detecting anomalies in record counts across pages and triggering the circuit breaker; or running a post-fetch reconciliation pass after a complete paginated result set has been assembled.
39. **API Version Management:** The connector configuration must declare the API version being targeted. The engine must support running different datatypes within the same connector against different API versions during a transition period. When a targeted API version approaches its deprecation deadline, the tool must emit warnings rather than failing silently after the fact.
40. **High-Water Mark Storage:** Incremental sync depends on a durable, per-connector per-datatype high-water mark (e.g., the last observed `updated_at` value or API change cursor). This value must be persisted in a dedicated watermark table in PostgreSQL — not in memory, not embedded in the connector config. The watermark must be updated atomically with the data write in the same database transaction, ensuring a crash never produces a watermark that is ahead of the data actually written. Operators may reset a watermark to a specific value or to epoch zero via the runtime control table to trigger a partial or full re-sync.
41. **Soft-Delete Resurrection:** If a record has been tombstoned (per requirement #32) and the external system later creates a new object with the same `external_id`, the ingestion tool must detect this and transition the record back to active state — clearing the deletion flags and writing the new payload. The previous deletion metadata must be preserved in the history table (if append mode is active) for audit purposes.
42. **Webhook HTTP Server Hardening:** The persistent HTTP server for receiving inbound webhooks has a real attack surface and must be hardened beyond payload signature verification (#34). Requirements: TLS must be supported (via native configuration or a reverse proxy); endpoint-level inbound rate limiting must be enforced to prevent flooding; an IP allowlist per connector must be optionally configurable; the webhook receiver port must be independently configurable per deployment; health and readiness endpoints must be served on a separate port from the webhook receiver (see Cross-Cutting Concerns).
43. **Connector Validation Mode:** Before running a connector for the first time, operators must be able to validate it non-destructively: check config syntax, test connectivity to the source system, verify authentication, and perform a single-page dry-run fetch without persisting any data or making any database writes. Output must clearly report the result of each check. This mode is activated via CLI or the runtime control table.
44. **Source Unavailability Handling:** When an external source system is entirely unreachable (DNS failure, connection refused, all retries exhausted), the ingestion tool must: cease retry attempts for that connector after a configurable back-off ceiling; mark the connector as unhealthy in the runtime control table; emit an alert-level log event and increment a dedicated error counter; and continue processing all other connectors unaffected. Recovery must be automatic — when connectivity is restored the tool must resume without a manual restart.
45. **Timestamp & Timezone Normalisation:** External APIs return timestamps in widely varying formats — Unix epoch (seconds or milliseconds), ISO 8601 with or without timezone offset, RFC 2822, and others. The ingestion tool must normalise all ingested timestamps to UTC `timestamptz` before storing them in any column. The expected source format for each timestamp field must be declaratively configurable per connector. Normalisation failures must be logged and must never silently store an incorrect value.
46. **Multi-Connector Fan-In Policy:** Whether multiple connectors may write the same logical datatype into a single shared table (e.g., contacts from both HubSpot and Salesforce into one `contacts` table) must be explicitly declared. Default behaviour is one table per connector — no fan-in. Shared-table fan-in must be opted into explicitly in the config; when enabled, `external_id` must be scoped by a connector discriminator column to prevent key collisions across sources.

## Tool 2: The Synchronization Tool (Writeback)
**Objective:** Read refined data from PostgreSQL and synchronize it back into external HTTP APIs.

**Key Requirements:**
1. **Per-Datatype Mapping:** Map the MDM-produced relational table to its respective external system endpoint datatype using declarative configuration.
2. **Smart Writes:** The writeback should exclusively target records that have changed, prioritizing incremental pushes. If an API requires bulk full writes, it should handle that smoothly.
3. **Conflict Resolution & Prevention:** The external system may receive conflicting writes from other applications or users. The tool must provide strong conflict prevention (e.g., Optimistic Concurrency Control, pre-flight state checks, or HTTP conditional requests) before committing data.
4. **Base-Aware Updates:** The writeback input may include a "base" — the subset of properties representing the version of the entity the update was computed from. This enables three-way merge and precise conflict detection against the current state in the target system.
5. **Client-Side Patching:** When the target system supports partial updates (PATCH), the tool should support a lookup-diff-write cycle: fetch the current state, compute a minimal client-side diff against the desired state, and submit only the changed fields.
6. **CRDT Support:** If the target system supports CRDT (Conflict-free Replicated Data Type) structures (e.g., counters, sets, registers), the writeback tool should leverage those data structures to apply updates without conflicts.
7. **Desired-State Input Table:** The MDM produces a desired-state table per target datatype with the following structure:
   - **`action`** column: one of `insert`, `update`, `delete`, `archive`, or `noop`.
   - **`cluster_id`** column: the MDM merge-group identifier (origin identity).
   - **Primary key column(s):** the external system's identifier(s) for the record. Present for `update`, `delete`, and `archive` actions; absent for `insert` (only `cluster_id` is known before creation).
   - `noop` rows indicate the desired state is already applied and can be skipped.
   - **Row lifecycle:** Processed rows are not immediately deleted from the desired-state table. Instead they transition to a processed/archived state or are retained with a `_processed_at` timestamp as an audit trail. The retention period is configurable per datatype. The tool must never delete a desired-state row that has not yet been successfully processed.
8. **Identity Mapping:** When creating new objects (`insert`), the external system generates an ID. The tool must capture these generated IDs and persist the link between the originating `cluster_id` and the generated external identity in dedicated mapping tables, making them available for future operations. To prevent duplicate inserts from concurrent retries, the identity mapping table must enforce a unique constraint on `(cluster_id, connector, datatype)` at the database level, and this constraint must be checked before issuing the HTTP request. If an insert response is lost after the external system confirmed creation (network failure post-response), the tool must detect the potential duplicate on retry by first querying the target system for an existing record matching the `cluster_id` reference field before issuing a new insert.
9. **Last-Written-State Tables:** The tool must store and expose the last successfully written state of each record as queryable tables. This provides an audit trail and serves as the "base" for future diff computations. For partial failures — where a write partially succeeds (some fields accepted, others rejected by the target system) — the last-written-state captures only the confirmed successfully written state, not the intended desired state. Any discrepancy between desired state and last-written state must be flagged and logged for operator review.
10. **Near Real-Time Writeback:** The tool should listen for finalized changes produced by the MDM (e.g., via PostgreSQL triggers, logical replication, or event queues) and proactively push these updates to the external systems as soon as possible.
11. **Politeness & Rate Limiting:** While striving for near real-time updates, the writeback process must remain gentle and respect external system limits. It must enforce strict rate limiting, request batching/chunking, and exponential backoff to ensure we do not bombard the target with a high volume of concurrent write operations (avoiding DoS).
12. **API Asymmetry Handling:** The read (GET) and write (POST/PUT/PATCH) data models for the same datatype in an external system may differ significantly. The declarative config must support mapping between asymmetric read/write schemas so the tools can bridge this gap transparently — ensuring that what comes out of ingestion is structurally compatible with what goes into writeback.
13. **Response Capture & Audit Logging:** After every write operation (insert, update, delete), the tool must capture and persist the full HTTP response — status code, headers, and body — alongside the written record. This enables: capturing generated IDs from insert responses, providing a queryable audit trail of all outbound writes, and supporting troubleshooting by logging error codes and messages returned by the target system.
14. **Duplicate Insert Prevention / Idempotency Guards:** Inserts are inherently non-idempotent. If a write is retried due to a timeout, crash, or batch failure, the tool risks creating duplicate records in the target system. The tool must implement strong duplicate detection — verifying against the write log before every insert that the entity has not already been successfully delivered. Batch-level atomicity must also be considered: only failed records in a partially failed batch should be retried, never successfully written ones.
15. **Separate Processing Paths per Operation Type:** Inserts, updates, and deletes have fundamentally different semantics, failure modes, and idempotency characteristics. They must flow through logically separate processing paths to prevent cross-contamination. In particular, deletes must only be triggered for records genuinely marked as deleted, not for records replaced by a newer version with a different identity.
16. **External Reference Field Writeback:** When the target system supports an "external reference" or "external ID" field on its records, the writeback tool should populate it with the MDM's `cluster_id` or origin identifier. This creates a reverse link from the target system back to the MDM, enabling simpler re-linking on future ingestion without depending solely on internal identity mapping tables.
17. **Pre-Write Data Transformation:** Data arriving from the MDM may be in a normalized or canonical form that differs from the target API's expected payload structure. The writeback tool must support a declarative transformation step that reshapes data into the target system's schema before the write — adapting field names, restructuring nested objects, and applying type coercions as defined in the connector configuration.
18. **Datatype-Specific Operation Configuration:** Each datatype in the target system may require different HTTP methods, URL patterns, headers, and payload structures for insert, update, delete, and lookup operations. The declarative configuration must support per-datatype, per-operation definitions rather than falling back to a single generic template.
19. **Upsert Write Strategy:** Some target systems expose a single idempotent upsert endpoint that handles both insert and update in one call. The writeback tool must support `upsert` as an action type in the desired-state table, routing to the appropriate endpoint when the target supports it.
20. **Archive Action Type:** The `archive` action in the desired-state table triggers a soft-delete in the target system — marking the record as inactive or archived rather than destroying it — where the target API supports this distinction. Archive must flow through a separate processing path from hard `delete` and must never be triggered for records that have simply been superseded by a newer version with a different identity.
21. **Transaction-Level Atomicity:** When the writeback tool reads a batch of desired-state changes that originated from the same MDM transaction, it must treat them as an atomic unit where possible — processing all changes together and treating failures at the transaction level. A partial failure within a multi-record MDM transaction must not leave the target system in an inconsistent intermediate state. When the target API does not support multi-record atomic operations (the common case), the fallback is: process records in dependency order per requirement #26, write each record individually, and if any record fails, route all remaining records from the same transaction group to the dead-letter queue — preserving the transaction group ID so the full group can be replayed together after the issue is resolved.
22. **`REPLICA IDENTITY FULL` on Desired-State Tables:** The PostgreSQL desired-state tables must be configured with `REPLICA IDENTITY FULL` so that logical replication change events carry the full `before` row values alongside the `after` values. This is a prerequisite for base-aware 3-way merge (requirement #4), enabling the writeback tool to reconstruct what the desired state was before the latest MDM update.
23. **Pre-Write Payload Validation:** Before submitting a write to the target API, the constructed payload must be validated against the expected schema for that operation type and datatype — catching field mapping errors, missing required fields, and type mismatches before they reach the external system. Validation failures must be logged with sufficient detail to identify the offending field and value.
24. **Dead-Letter Queue for Permanently Failed Writes:** When a write operation fails and all configured retry attempts are exhausted, the record must be moved to a per-datatype dead-letter table rather than silently discarded or retried indefinitely. Dead-letter entries must include: the original desired-state record, the full error response, the timestamp of first failure, the retry count, and sufficient diagnostic context for an operator to identify and correct the underlying issue. Dead-letter entries must be replayable via the runtime control table once the issue has been resolved.
25. **Writeback Circuit Breaker:** If the target system returns a sustained pattern of errors (e.g., consecutive 500s or 503s), the writeback tool must enter a paused state for that connector rather than continuing to send failing requests. The tool must respect `Retry-After` headers on 429 responses. The breach threshold and pause duration must be configurable per connector. Circuit breaker state must be visible in the runtime control table and resettable by an operator.
26. **Dependency Ordering Within a Write Batch:** When a write batch contains records where some must exist in the target system before others can be created (e.g., parent accounts before child contacts, or association objects before their member records), the declarative configuration must support expressing these ordering dependencies. The writeback tool must topologically sort records within a batch before submission. If the declared dependency graph contains a cycle, the entire affected dependency group must be halted — no records in the group are submitted — and the cycle must be reported as a config error in both the runtime control table and logs. Cycles are never resolved heuristically.
27. **Dry-Run / Preview Mode:** The writeback tool must support a dry-run mode in which all write logic executes — conflict detection, payload construction, validation, ordering — but no HTTP requests are sent to the target system. Instead, the tool logs or stores exactly what requests would have been made (method, URL, headers, body) alongside the expected outcome. This mode is essential for validating new connectors, debugging transformation logic, and reviewing the impact of a first writeback run. It is activated via the runtime control table.
28. **Write Ordering for the Same Record:** If multiple desired-state updates for the same external record enter the writeback queue in rapid succession, they must be processed strictly in the order they were produced. A newer update must never be superseded by an older one. The tool must detect and discard or demote obsolete queued writes for a record when a newer write for the same record is already in flight or has been processed.
29. **Partial-Success Batch Response Handling:** Some external APIs accept batch write requests and return mixed success/failure responses — for example, an HTTP 200 or 207 with a body enumerating per-record outcomes. The writeback tool must declaratively parse such responses to correctly classify per-record success and failure, retry only the failed records, and record confirmed successes. Retrying the entire batch on a mixed response is not acceptable.
30. **Conflict Detection — Resolution Path:** When a conflict is detected (the target system's current state differs from the expected base, per requirement #3), the tool must apply a declared resolution strategy rather than failing silently or halting. Available strategies per connector/datatype: `dead-letter` (route the conflicting write to the dead-letter queue for operator review — the default), `last-writer-wins` (overwrite regardless of conflict), or `skip-and-warn` (discard the write and emit a warning metric). Regardless of strategy, every detected conflict must be logged and counted as a dedicated observability metric.
31. **Delete Safety Guard:** Before processing a `delete` or `archive` action, the writeback tool must verify that the record still exists in the target system and that its current state matches the last-written state known to the tool. If the record has been externally modified since the last write, the delete must not proceed and must be routed through the conflict resolution path (requirement #30). If the record no longer exists in the target system, the delete is a no-op — it must be logged and marked as successfully processed.
32. **Replication Slot Health Monitoring:** The logical replication slot used by the writeback trigger path (#10, #22) accumulates WAL if the consumer falls behind. An unchecked stalled slot can exhaust disk and crash PostgreSQL. The writeback tool must: continuously monitor replication slot lag (bytes behind and estimated time behind); emit this as an observability metric; log at ERROR level and increment a dedicated counter when lag exceeds a configurable warning threshold; and support a fallback mode where, if lag exceeds a configurable maximum threshold, the tool pauses replication consumption and falls back to polling the desired-state table directly until the backlog clears.
33. **Write Batch Composition:** The mechanism by which records are assembled into write batches must be explicitly defined rather than implementation-dependent. Batches are closed by whichever threshold is reached first: (a) a configurable maximum record count, (b) a configurable maximum uncompressed payload size in bytes, or (c) a configurable maximum wait time (age of the oldest record in the forming batch). Ordering within a batch must respect requirement #28 (write ordering per record) and requirement #26 (dependency ordering). Batch composition parameters must be configurable per connector per operation type.
34. **Cluster Merge & Split Propagation:** MDM identity resolution can reassign records between clusters — merging two clusters into one (many-to-one), or splitting one cluster into multiple (one-to-many). The desired-state table must support a `merge` action (specifying the losing `cluster_id` and the surviving `cluster_id`) and a `split` action (specifying the originating `cluster_id` and the resulting new `cluster_id` values). For `merge`: the writeback tool must update the target record with the surviving identity and delete or archive the losing record, then update identity mapping tables. For `split`: the tool must create new target records for each child cluster. These actions must flow through the same safety-guard checks as `delete` and `insert` respectively.

## The MDM Contract

Both tools communicate exclusively through PostgreSQL — they never call each other directly. The ingestion tool writes into a well-defined set of PostgreSQL tables. The MDM layer (outside the scope of these tools) reads from ingestion tables, applies identity resolution, deduplication, and enrichment logic, and writes the results into desired-state tables. The writeback tool reads those desired-state tables and pushes changes to external systems.

**Ingestion output contract:** One source-of-truth table per datatype, containing `external_id`, `data` (JSONB), `raw` (JSONB), and the standard metadata columns defined in requirement T1 #2. A separate history/audit table is present when `append` mode is configured.

**Writeback input contract:** One desired-state table per target datatype, with at minimum `action`, `cluster_id`, `data` (JSONB desired payload), and the primary key column(s) as defined in the connector config.

**Shared support tables:** A sync-run log (common to both tools), the runtime control table, identity-mapping tables (owned by the writeback tool, readable by ingestion for reverse-lookup), the credential store, and the dead-letter tables (one per datatype per tool, owned by the respective tool). Neither tool reads from the other's output tables directly — the MDM layer is the sole intermediary.

**Sync-run log schema:** The sync-run log table must include at minimum: `run_id` (UUID PK), `connector`, `datatype`, `mode` (`full` or `incremental`), `started_at`, `finished_at`, `status` (`running`, `completed`, `failed`, `aborted`), `records_fetched`, `records_written`, `records_errored`, `records_skipped`, `high_water_mark_before`, `high_water_mark_after`, and `error_detail` (JSONB, nullable). Both tools write to this table. It is the primary operational record of what ran, when, and with what outcome.

**Schema migration coordination:** When a new version of either tool changes the schema of any managed table, the migration must be performed explicitly and coordinated with the MDM layer. Both tools must enforce a schema-version check at startup: if the database schema version does not match the tool's expected version, the tool must refuse to start and emit a clear diagnostic error. Migrations are never applied automatically at startup — they must be run explicitly by the operator via a dedicated migration command.

## Cross-Cutting Concerns

### Error Classification & Retry Policy

Both tools must apply a consistent taxonomy when classifying errors and deciding on retry behaviour:

| Class | Examples | Behaviour |
|---|---|---|
| Retryable transient | Network timeout, 503, connection reset | Exponential backoff with jitter; configurable max retries |
| Rate limit | 429 with `Retry-After` | Honour `Retry-After`; pause the relevant connector/datatype |
| Auth error | 401, expired token | Attempt credential/token refresh once; if it fails, escalate to dead-letter |
| Data / validation error | 422, 400, schema mismatch | Not retryable; log with full detail and move to dead-letter immediately |
| Config error | Missing required field, unknown endpoint | Halt all activity on the affected connector and alert; retrying will never help |

Per-connector overrides for retry limits and backoff parameters must be supported in declarative config.

### Observability & Alerting

Both tools must export the following minimum set of operational signals:

- **Sync lag per datatype:** time elapsed since the last successful sync for each datatype/connector pair.
- **Records processed / skipped / errored per run:** counters broken down by outcome.
- **HTTP error rates:** by status code, connector, and datatype.
- **Queue depth / backlog:** number of pending desired-state rows awaiting processing (writeback).
- **Replication slot lag:** WAL bytes behind for the logical replication trigger path (writeback).
- **Circuit breaker state:** open/closed/half-open per connector and per datatype.
- **Dead-letter queue depth:** unresolved permanently failed records per datatype.

Metrics must be exportable in Prometheus-compatible format. Structured JSON logs must include `sync_run_id`, `connector`, and `datatype` on every log entry. Each sync operation must emit distributed trace spans compatible with OpenTelemetry.

### Data Privacy & PII

MDM systems by nature handle personally identifiable information. Both tools must:

- **Never log raw field values at INFO level.** Field-level masking and suppression must be declaratively configurable per connector — specific fields can be hashed or redacted in logs and audit tables.
- **Support targeted purge by `external_id`:** tombstone records, last-written-state entries, dead-letter entries, and audit logs must all support deletion of a specific individual's data to comply with GDPR right-to-erasure and equivalent regulations.
- **Annotate PII fields in the connector schema** so downstream data governance tooling can discover and classify sensitive fields without inspecting raw data.

### Retention & Data Housekeeping

Operational tables will grow without bound unless actively managed. Both tools must support configurable retention policies for: sync-run log entries, dead-letter entries, last-written-state rows, desired-state processed rows, and append-mode history table rows. Retention is expressed as a maximum age (e.g., `90d`) or a maximum row count per datatype — whichever is more restrictive. A scheduled housekeeping job (configurable interval, activatable via the runtime control table) purges rows beyond the retention threshold. Housekeeping must never delete rows still required by active processing (unprocessed desired-state rows, open sync-run records). Retention policy must co-exist cleanly with GDPR targeted-purge requirements — a targeted purge by `external_id` takes immediate precedence over any retention window.

### Circuit Breaker Lifecycle

Both tools use circuit breakers (T1 #13, T2 #25). The half-open state is operationally defined as follows: after a circuit breaker has been open for the configured pause duration, it automatically transitions to half-open and sends a single probe request to the external system. If the probe succeeds, the breaker closes and normal processing resumes. If the probe fails, the breaker re-opens and the pause duration resets (with optional exponential back-off on successive open cycles). The probe count before declaring recovery, and the back-off multiplier on repeated opens, are configurable per connector. Every circuit breaker state transition (closed → open, open → half-open, half-open → closed, half-open → open) must emit a structured log event and increment a dedicated observability counter.

### Health & Readiness Endpoints

Both tools must expose two standard HTTP endpoints for integration with container orchestration (Kubernetes, etc.) and load balancers:

- **`GET /health`** (liveness): Returns HTTP 200 while the process is alive and capable of self-recovery. Returns non-200 only for unrecoverable states. Must respond within 1 second.
- **`GET /ready`** (readiness): Returns HTTP 200 when the tool has completed startup (config loaded, DB connection established, replication slot attached if applicable) and is actively processing work. Returns non-200 during startup, graceful-shutdown drain, or when the tool has self-flagged as unhealthy. Must include a minimal JSON body indicating which connectors are active, paused, or in circuit-breaker state.

Both endpoints must be served on a dedicated internal port, separate from the webhook receiver port, so they are not reachable from the same network path as inbound external webhooks.

2. **Database Architecture:** Design the PostgreSQL schema utilizing `JSONB` for raw payloads and metadata columns for sync state tracking.

Define the contract between the engine and simulators so any connector can be exercised without live credentials.
4. **Ingestion Engine:** Build the extractor that reads configs, fetches HTTP data (incremental/full), and writes to the DB.
5. **Writeback Engine:** Build the synchronizer that reads from the DB and pushes to external HTTP APIs with conflict resolution.

The CLI must connect directly to the PostgreSQL instance managed by the daemons.
7. **Connector SDK & Documentation:** Define and publish the connector authoring contract — the configuration schema specification, the simulator interface contract, the required test-suite structure, and the metadata annotations (PII fields, operation definitions, dependency declarations). Produce a reference connector implementation with an accompanying simulator to serve as the canonical example for future connector authors.
